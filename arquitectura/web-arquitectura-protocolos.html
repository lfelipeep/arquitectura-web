HTTP — Puerto 80

Protocolo de Transferencia de Hipertexto. Texto plano, sin cifrado. Hoy en desuso en producción.

HTTPS — Puerto 443

HTTP + TLS/SSL. Todo el tráfico va cifrado. Es el estándar obligatorio en la web moderna.

¿Qué pasa exactamente? El cliente envía un HTTP Request (método + URL + headers + body opcional). El servidor procesa la petición, accede a la base de datos si es necesario, y devuelve un HTTP Response (código de estado + headers + body con datos).

Verbos HTTP

Los métodos que definen qué acción se quiere realizar sobre un recurso.

GET

Leer / Obtener

Trae datos del servidor. No modifica nada. Idempotente y sin body.

POST

Crear recurso

Envía datos al servidor para crear un nuevo recurso. Lleva body con JSON.

PUT

Reemplazar completo

Reemplaza un recurso entero. Debes enviar todos los campos del objeto.

PATCH

Actualización parcial

Modifica solo algunos campos de un recurso existente. Más eficiente que PUT.

DELETE

Eliminar recurso

Borra el recurso indicado en la URL. Generalmente sin body en la petición.

Ejemplo CRUD con una API de usuarios
GET /api/users → listar todos los usuarios GET /api/users/42 → obtener usuario con id=42 POST /api/users → crear un usuario nuevo PUT /api/users/42 → reemplazar usuario 42 PATCH /api/users/42 → actualizar email del usuario 42 DELETE /api/users/42 → eliminar usuario 42

Códigos de Estado HTTP

El servidor siempre responde con un código numérico que indica el resultado de la petición.

2xx — Éxito
200 OK Petición exitosa. El servidor devolvió los datos solicitados.
201 Created Recurso creado correctamente. Típico de POST exitoso.
4xx — Error del cliente
400 Bad Request La petición tiene sintaxis incorrecta o datos inválidos.
401 Unauthorized No autenticado. Necesitas iniciar sesión o enviar un token.
403 Forbidden Autenticado pero sin permiso. No tienes acceso a este recurso.
404 Not Found El recurso solicitado no existe en el servidor.
5xx — Error del servidor
500 Internal Server Error Error inesperado en el servidor. El problema es del backend.

Headers HTTP Importantes

Metadatos que viajan en cada petición y respuesta para comunicar formato, autenticación y más.

Headers de Request
Header Valor ejemplo Para qué sirve
Content-Type application/json Indica el formato del body que se envía al servidor.
Authorization Bearer eyJhbGci... Envía el token JWT para autenticación.
Accept application/json Le dice al servidor qué formato de respuesta espera el cliente.
User-Agent Mozilla/5.0... Identifica el cliente (navegador, app) que hace la petición.
Headers de Response
Header Valor ejemplo Para qué sirve
Content-Type application/json; charset=utf-8 Indica el formato del body de la respuesta.
Status 200 OK Código de estado de la respuesta.
Cache-Control max-age=3600 Indica cuánto tiempo puede el cliente cachear la respuesta.
Access-Control-Allow-Origin * Política CORS — qué dominios pueden acceder a la API.

URLs, URIs y Parámetros

La anatomía de una URL y la diferencia entre path params y query params.

Anatomía de una URL completa
https://api.ejemplo.com/users/42/posts?page=2&limit=10#comentarios
https:// → Protocolo api.ejemplo.com → Dominio /users/42/posts → Path (incluye path param: 42) ?page=2&limit=10 → Query Params #comentarios → Fragment

Path Parameters

/users/:id
  • Identifican un recurso específico
  • Van dentro de la ruta de la URL
  • Son obligatorios para la petición
  • Ejemplo: /products/99

Query Parameters

?key=value
  • Filtran, ordenan o paginan resultados
  • Van después del ? en la URL
  • Son opcionales generalmente
  • Ejemplo: ?sort=name&page=3
URI vs URL: Una URI (Uniform Resource Identifier) es el identificador genérico de cualquier recurso. Una URL (Locator) es una URI que además indica cómo acceder al recurso (protocolo + ubicación). Toda URL es una URI, pero no toda URI es una URL.

SPA vs MPA

Dos arquitecturas frontend con ventajas distintas según el tipo de aplicación.

SPA

Single Page Application
  • Carga una sola página HTML al inicio
  • Navega sin recargar el navegador
  • Renderizado en el cliente (JS)
  • Mejor experiencia de usuario fluida
  • Más complejo para SEO
  • Ejemplos: React, Vue, Angular apps

MPA

Multi Page Application
  • Cada ruta carga una página nueva del servidor
  • El servidor renderiza el HTML completo
  • Recarga completa en cada navegación
  • Excelente para SEO por defecto
  • Más simple de implementar
  • Ejemplos: sitios en WordPress, Django, Rails
Flujo de navegación comparado
SPA: Usuario hace clic → JS intercepta → fetch() → actualiza DOM parcial Sin recarga, estado conservado, transición instantánea MPA: Usuario hace clic → navegador envía GET → servidor devuelve HTML completo Recarga completa, nueva página desde cero cada vez
